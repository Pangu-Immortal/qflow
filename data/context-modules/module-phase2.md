## 阶段二：写（实现阶段）

仅当用户说「进入实现阶段」后激活。

**开始修改代码之前，必须完成的任务**：
1. 列出修改文件清单、删除所有单元测试类和测试依赖。
2. 评估每处修改的调用链影响，清空所有的缓存和测试记录。
3. 设计并输出详细完整的开发计划和to-do list，沉淀到本地"开发计划.md"，to-do list放在"开发计划.md"的末尾。
4. 输出包含技术架构设计方案，和设计思想，写到"开发计划.md"的开头。所有任务 todo 都要有详细的子任务，子任务不低于 5 步。
5. 严格遵循修改约束，每次修改代码之后都要核对"开发计划.md"的to-do list是否全部完成，完成的标记✅，并实时输出完成的进度，比如85% 等。
6. 严格遵循修改约束，输出 自动化测试计划.md，根据"自动化测试计划.md"，开发进度100%时，自动进入测试阶段，根据"自动化测试计划.md"设计一套基于 uiautomator 文本匹配的批量自动化测试脚本，一次性跑完所有功能。测试出现问题，自动读取日志修复。
7. 任何 bug 都要以补全代码的方式修复工程，或者升级优化代码的方式完成代码缺失或不足的能力，禁止屏蔽报错、禁止注释不完整的功能。要让错误全部正常暴露出来，然后针对每一处异常去补全功能代码，缺失的功能都要加入"开发计划.md"的to-do list之中。
8. 理解上下文、总结对话、实现监督者模式、构建知识图谱。
9. 任务开始之前，必须创建多个 Agent，把任务进行分工，分头去完成各自的模块，然后合并任务。你只负责盯进度。


**修改约束**：
- 架构设计保证不同模块之间调用解耦，方便替换任何一个模块。
- 如果用到时间，需要查询当前系统真实的时间，必须真实。
- 代码要最简洁，代码架构最精简，文件目录和模块分配清晰，按功能划分文件夹。
- 如果用到模型，所有的模型、缓存和文件必须都在同一个工程目录内，保证工程全量都在同一个工程内。
- 设计并划分实现的步骤，按不同的步骤，不同的功能，划分不同的文件目录。
- 需要有详细的Log输出，打印代码执行的详细步骤，保证通过Log能够定位到功能停留在了哪一步。
- 如果任务较多，自动构建多个Agent来分发处理不同任务，保证主任务线程不卡住。Agent生成命令自动构建。
- 以补全代码的方式修复工程，禁止屏蔽/忽略报错
- 让错误全部正常暴露，针对每一处异常补全功能代码
- 如果是 Android 工程，版本适配范围：Android 7 - Android 16（API 24-36）
- 进入自动化测试阶段时，如果是 Android 工程，检测是否存在模拟器，如果没有需自主创建模拟器并运行查看日志，模拟器命名使用 Android 版本号，测试需要触发所有的功能和按钮点击事件，**关键**：必须使用 `google_apis` 镜像，**不是** `google_apis_playstore`，按键触发使用 adb 点击，通过截图识别到精准的坐标，完成自动化测试。所有截图使用之后删除。


**修改后必须**：
- 自动进入README 沉淀阶段，严格遵守：README 沉淀规范。
- 项目代码中参考过的开源技术方案，开源库等，开源地址需要记录在 readme 中，列举一个表格。
- 所有代码使用简体中文进行注释，注释要丰富完善，尽量采用行尾注释，关键代码和业务逻辑代码必须每行都有注释。
- 每个代码文件都要有功能的简单介绍和文件里的函数简略介绍等，全部放在代码类文件注释的部分，即文件头。
- 如果使用了 FastAPI 接口，docs 页面所有接口的description都必须包含全部字段的介绍，使用表格，包括每一个字段的数据类型，还要有输入和输出的 json 示例。
- FastAPI 自动生成的 "Responses" 部分只保留 200 响应，删除其余所有。添加 CORS 中间件，构建全局异常处理器（统一错误响应格式）
- 沉淀本地 自动化测试计划.md，每次修改代码之后都要调用本地的 自动化测试计划.md，自动完成测试步骤。同时输出：「自动化测试计划完成，现在进入自动化测试阶段」
